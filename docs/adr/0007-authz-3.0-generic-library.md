# ADR-0007 — Lumen.Authorization 3.0: generic authorization library

**Status:** Accepted
**Date:** 2026-07-14
**Supersedes:** ADR-0004 (discovery/sync/reconciliation/catalog sections only)
**Related:** SPEC-0001, ADR-0005, ADR-0006

---

## Context

Lumen.Authorization 2.x shipped with an automatic permission catalog machine: a `PermissionDiscoveryScanner` that scanned `[RequirePermission]`-decorated actions at boot, a `PermissionSyncService` that upserted permissions and groups in the database, a `PermissionCatalogValidationService` that warned about missing codes, and a `LumenAuthorizationStartupService` that orchestrated all of the above. The `Permission` entity carried `Controller`, `Action`, `IsOrphan`, and `OrphanedAt` columns to support this workflow.

The goal was developer convenience (zero manual seeding). In practice it created coupling: the library had opinions about application code, auto-modified the consumer's catalog at every boot, and required `PermissionCatalogMode` options that were rarely understood. It also left the schema polluted with columns that only served the auto-catalog path.

No consumer reached production before this revision.

---

## Decision

Remove the entire automatic catalog machine (discovery, sync, reconcile, validation, orchestrator) and declare Lumen.Authorization a **generic, zero-opinion authorization library** per the four inviolable principles in SPEC-0001:

1. **Zero auto-population.** Tables start empty. The library creates the schema, nothing else.
2. **Zero global enforcement.** The library only blocks where `[RequirePermission]` is explicitly placed.
3. **Zero identity coupling.** No `Users` table. `userId` is an opaque Guid from a configurable claim.
4. **Zero consumer-specific code.** The library knows nothing about any application.

**Schema changes (breaking):**
- Drop `Permission.Controller`, `.Action`, `.IsOrphan`, `.OrphanedAt`.
- `Permission.Create` now receives `(code, displayName, groupPermissionId?)`.
- Migrations regenerated from scratch (no ALTER chain — pre-production).

**API surface changes (breaking):**
- `LumenAuthorizationOptions`: remove `CatalogMode`, `FailFastOnMissingPermission`, `AutoGrantAllToAdministrator`.
- Remove `IPermissionSyncService`, `PermissionSyncService`.
- Remove `PermissionDiscoveryScanner`, `PermissionDiscoveryAndReconciliationHostedService`, `PermissionCatalogValidationService`, `LumenAuthorizationStartupService`.
- Remove `PermissionCatalogMode` enum.
- `AddLumenAuthorization` umbrella: core + enforcement + migrations only.

**Additions:**
- `LumenBackofficePermissions` public constants (`ProfilesView`, `ProfilesManage`, `PermissionsView`, `PermissionsManage`, `GroupsManage`, `UserProfilesManage`) so consumers can seed and assign backoffice access.
- `SeedLumenPermission*` helpers corrected: singular table names, `description` column on group, no dead columns.

**Maintained:** `RequirePermissionAttribute`, enforcement pipeline, `IUserPermissionService`, caching, CQRS for profiles/permissions/groups/user-profiles, `IUserDirectory`, `ITenantScopeAccessor`, `GetAuthorizationGraphQuery` (field `Orphan` removed from result).

---

## Consequences

**Positive:**
- Library is genuinely generic: zero app-specific behavior.
- Schema is clean: no dead columns.
- Startup is faster: no catalog scan on every boot.
- Consumer owns the catalog completely via migrations.

**Negative/Trade-offs:**
- Consumers must seed permissions explicitly (migration helpers provided).
- No auto-detection of "you forgot to seed this permission" (was `FailFastOnMissingPermission`). A missing code simply means nobody has it — 403, never a boot failure.
- Consumers must decide who administers `/lumen` by seeding and assigning `LumenBackofficePermissions.*` codes.

---

## Migration (2.x → 3.0)

See SPEC-0001 §9 for the complete guide. Summary:
1. Remove `CatalogMode`/`FailFastOnMissingPermission`/`AutoGrantAllToAdministrator` from `AddLumenAuthorization` call.
2. Drop and re-create the `Lumen` schema (dev environment only — not in production).
3. Create a consumer migration using `SeedLumenPermissionGroup` / `SeedLumenPermission`.
4. Seed `LumenBackofficePermissions.*` codes and assign them to the admin profile.
